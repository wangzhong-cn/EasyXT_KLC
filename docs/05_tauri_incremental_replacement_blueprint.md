# EasyXT Tauri 增量替换执行蓝图

> 目的：把“继续在 PyQt 内部深化”切换为“建立可观察、可验证、可渐进迁移的前端与状态边界”。

**版本**: v0.1
**最后更新**: 2026-03-31
**状态**: ✅ 当前默认执行路线

---

## 为什么要切换路线

当前系统的主要问题已经不是“功能是否存在”，而是：

- 前端无法稳定观察后台行为
- 用户无法在真实工作流中验证系统状态
- PyQt 内部线程、面板和局部状态太多，验证成本过高
- API、Bayesian、结构详情等能力虽然已经存在，但主要仍在旧壳中消费

因此，本阶段的目标不是继续扩展 PyQt 交互，而是建立**可观察边界**。

一句话原则：

> **先恢复可观察性，再继续做功能。**

---

## 当前默认架构判断

### 默认状态

- **Tauri**：默认增量替换前端壳
- **SQLite (WAL)**：默认可写主状态
- **DuckDB**：默认只读分析引擎
- **零运维演进**：SQLite 分片 + DuckDB 多文件联邦查询
- **影子文件原子同步**：数据一致性的最后保险，属于架构铁律

### 本阶段不做的事情

- 不继续优先深化 PyQt-only 交互
- 不把 DuckDB 继续用作默认在线可写状态库
- 不引入 Docker / PostgreSQL / TimescaleDB 依赖
- 不先做“大一统重写”

---

## P0 替换范围

第一批必须迁入 Tauri 的不是“最花哨的页面”，而是“最需要被验证的页面”。

### P0 页面

1. **结构列表页**
   - 结构查询
   - 筛选/分页
   - Layer 4 可见
2. **结构详情页**
   - 单结构详情
   - 最新信号
   - 审计时间线
   - posterior/group 元信息
3. **Bayesian 操作页 / 面板区**
   - preview
   - apply
   - group_strategy / min_observations
   - 执行结果与失败原因
4. **同步状态页**
   - SQLite 当前版本
   - shadow 版本
   - DuckDB 当前只读版本
   - last_good_version
   - 同步滞后/失败原因

### P0 成功标准

用户必须可以在前端直接完成以下验证：

- 我发起了什么动作
- 这个动作命中了什么请求
- 这个请求写入了哪个状态版本
- DuckDB 当前读到的是哪个影子版本
- 如果失败，失败在什么阶段

## 增量替换原则

### 原则 1：壳替换优先于逻辑替换

先替换：

- 可观察层
- 请求层
- 状态层
- 路由层

后替换：

- 旧 Widget 的深层细节逻辑
- 复杂 PyQt 面板行为

### 原则 2：读优先，写收敛

- 先把**读场景**迁到 Tauri
- 再把**写场景**统一收敛到 SQLite WAL
- 最后才考虑更复杂的数据演进

### 原则 3：旧壳冻结，不再扩张

PyQt 进入以下状态：

- 可修复 bug
- 可保兼容
- 可临时托管旧功能
- **不再作为未来主要交互宿主**

## 分阶段执行计划

## Phase 0：冻结与立规

### Phase 0 目标

建立“以后该怎么干”的默认约束。

### Phase 0 输出物

- 本蓝图文档
- 双引擎状态契约文档
- 前端可观察性验收清单
- 明确 legacy / new shell 边界

### Phase 0 验收

- 仓库文档明确写出默认路线
- 新开发不再以“继续深化 PyQt”为默认方向

---

## Phase 1：Tauri P0 可观察前端

### Phase 1 目标

先做出一个最小但能验收真实行为的前端壳。

### Phase 1 范围

- 结构列表
- 结构详情
- Bayesian preview/apply
- 同步状态页

### Phase 1 必须具备的 UI 能力

- 请求可见
- 状态可见
- 错误可见
- 版本可见
- 最近操作可追溯

### Phase 1 验收

用户无需打开 PyQt 或读日志，也能判断：

- 当前系统读的是什么
- 写的是什么
- 为什么失败
- 是否已经同步到 DuckDB 只读影子

---

## Phase 2：SQLite (WAL) 主状态落地

### Phase 2 目标

把“谁是权威写源”定下来。

### Phase 2 范围

SQLite 负责：

- UI 状态
- 任务状态
- 操作记录
- 结构/Bayesian 执行状态
- 同步元数据
- 版本信息

### Phase 2 验收

- 新写路径默认进入 SQLite
- 前端状态不再依赖 PyQt 内存对象作为事实源
- 关键操作具备事务边界

---

## Phase 3：DuckDB 只读影子化

### Phase 3 目标

让 DuckDB 成为稳定、可回滚、只读的分析模型。

### Phase 3 范围

DuckDB 负责：

- 结构查询
- Layer 4 summary
- 只读宽表分析
- 历史回放与实验查询

### Phase 3 验收

- 读侧只读取“已提交的影子版本”
- 不再暴露半同步状态
- 支持 last_good_version 回退

---

## Phase 4：零运维横向扩展

### Phase 4 目标

在不引入额外运维负担的前提下，把 SQLite + DuckDB 扩展到更大单机容量。

### Phase 4 路线

- SQLite 按时间/标的进行应用层分片
- DuckDB 通过多文件 `ATTACH` / 联邦查询聚合相关分片
- 通过 shard catalog 与 manifest 控制查询范围
- 继续保持“文件即数据库、复制即备份”的零运维特性

### Phase 4 当前结论

- **完全排除 TimescaleDB / Docker 路线**
- 扩展性问题优先用应用层分片解决，而不是数据库层分布式解决

---

## 前后端职责边界

## 前端（Tauri）负责

- 可观察 UI
- 请求编排
- 状态显示
- 用户操作反馈
- 验证链路呈现

## 服务层负责

- 请求校验
- 状态写入 SQLite
- 影子同步调度
- DuckDB 只读查询
- 版本/同步状态暴露

## Legacy PyQt 负责

- 兼容旧流程
- 托管暂未迁移功能
- bug fix / 稳定性维护
- 不再优先新增复杂交互

---

## 开发停止线

以下事项在没有明确例外批准前，默认不再优先推进：

1. 新的 PyQt 深层联动工作流
2. 更多只存在于 Widget 内部的状态行为
3. 继续把 DuckDB 当作在线事务写主库
4. 未定义版本/同步状态就继续扩前端功能

---

## 每个新功能的验收四联单

以后每个新功能至少要回答：

1. **用户操作是什么？**
2. **写入的权威状态在哪里？**
3. **读侧何时看到它？**
4. **前端如何验证这一切？**

如果答不出来，就不应继续往前推进。

---

## 当前建议的第一批实施项

### 立即开始

- 建立 Tauri 工作区 / 壳项目
- 为结构列表 / 详情 / Bayesian / 同步状态设计路由与 API 消费层
- 定义 SQLite 主状态 schema
- 定义 shadow 同步 manifest 与版本号规则

### 暂缓

- PyQt 继续细化交互
- Docker / PostgreSQL / TimescaleDB 接入
- 复杂多进程前端扩展

---

## 完成定义

当以下条件满足时，本阶段可视为成功：

- 用户主要通过 Tauri 页面验证结构与 Bayesian 行为
- SQLite 成为默认主写状态
- DuckDB 成为默认只读影子
- 前端可直接显示同步版本、失败阶段和当前读模型版本
- PyQt 不再是主验证入口

---

## 一句话总结

> **EasyXT 当前阶段的默认执行路线，不再是继续深化 PyQt，而是以 Tauri 增量替换恢复可观察性，以 SQLite (WAL) 建立主状态，以 DuckDB 提供只读分析视图，并用影子文件原子同步保证一致性。**
